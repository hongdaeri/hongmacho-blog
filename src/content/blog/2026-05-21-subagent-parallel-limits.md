지난주 외주 프로젝트에서 백엔드 API 5개를 하루 안에 뽑아야 했다. ultrawork 패턴으로 executor 서브에이전트 5개를 동시에 띄웠다. 결과는 반반이었다. 속도는 기대 이상이었지만, 두 가지 문제가 터졌다.

## 실험 조건

5개를 동시에 띄운 데는 단순한 이유가 있었다. 48분짜리 작업을 10분 안에 끝낼 수 있다고 생각했다. 수학적으로 5개 병렬이면 1/5이니 약 10분. 마감이 그날 오후 6시였고 오전 10시에 시작했으니 여유가 있었다. 심지어 에이전트 스핀업 비용과 토큰 사용량까지 계산했다. 순차 대비 토큰이 1.6배 늘어도 시간당 비용 대비 이득이라는 판단이었다. 그 계산 자체는 맞았다. 문제는 계산에 없던 것들이 터졌다는 거다.

작업은 독립적인 REST API 엔드포인트 5개 구현이었다. 각 에이전트에게 엔드포인트 하나씩을 할당하고, 공통 유틸리티는 `src/utils/` 폴더에 이미 있다고 알려줬다. 프롬프트에는 파일 경로와 함수 시그니처를 명시했다.

wall-clock 기준으로 순차 실행 예상 시간은 약 48분. 병렬 5개를 띄웠을 때 실제 완료까지 걸린 시간은 11분이었다.

## 좋은 점 — wall-clock 1/4, 토큰 1.6배

속도는 확실히 압도적이다. 11분이라는 숫자는 순차 대비 약 77% 단축이다. 수학적으로 완전 병렬이면 1/5이 되어야 하는데 1/4에 그친 이유는 에이전트 스핀업 시간과 공통 파일 읽기 경합이 있었기 때문이다.

토큰 사용량은 순차 기준의 약 1.6배였다. 각 에이전트가 독립적인 컨텍스트 창을 가지기 때문에 공통 코드베이스를 5번 중복 읽는다. 비용은 높아지지만 속도와 교환하는 거래다. 외주 마감이 있는 상황에서는 수용 가능한 교환이었다. 시간당 단가로 환산하면 병렬 실행의 토큰 추가 비용보다 마감 지연 리스크가 훨씬 컸다. 그 계산은 지금도 유효하다. 다만 그 계산이 병합 비용을 포함하지 않았다는 게 문제였다.

## 문제 1 — 같은 파일 영역 충돌

가장 먼저 터진 건 `src/middleware/auth.ts` 파일이었다. 에이전트 2번과 4번이 각자 인증 미들웨어를 수정했다. 둘 다 기존 파일을 읽고 자기 방식대로 덮어썼다. 나중에 완료된 에이전트의 변경이 먼저 완료된 에이전트의 변경을 지웠다.

이 충돌은 예상해야 했는데 못 했다. 작업 명세를 "독립적인 엔드포인트"로 짰지만, 둘 다 인증이 필요한 엔드포인트였고 자연스럽게 같은 파일에 손을 댔다. "독립적"이라는 단어는 비즈니스 로직 수준의 독립을 말했는데, 에이전트는 파일 시스템 수준에서 독립이 아니었다. 그 둘을 구분하지 않은 게 실수였다.

## 문제 2 — 컨텍스트 누락, 중복 구현

에이전트 1번이 `src/utils/pagination.ts`에 페이지네이션 헬퍼를 만들었다. 에이전트 3번은 그 사실을 모르고 자기 엔드포인트 파일 안에 비슷한 로직을 인라인으로 구현했다. 결과물을 합쳐보니 페이지네이션 로직이 두 군데 존재했다.

더 복잡했던 건 `src/utils/index.ts`에서 터진 충돌이었다. 에이전트 2번과 5번이 각자 날짜 포맷 헬퍼를 만들었는데, 함수 이름이 달랐다. 하나는 `formatDateKST`, 다른 하나는 `toKSTString`이었다. 두 함수 모두 같은 일을 하는데 이름이 다르고 구현 방식도 미묘하게 달랐다. 어느 쪽이 맞는 버전인지 알 수 없었다. 각 에이전트의 엔드포인트가 서로 다른 함수를 호출하고 있었기 때문에 하나를 지우면 그쪽 엔드포인트가 깨졌다. 어느 쪽을 표준으로 삼고 호출부를 일괄 변경하는 데 30분이 걸렸다. 속도로 아낀 시간을 병합 단계에서 고스란히 돌려줬다.

병렬 에이전트는 서로의 작업 결과를 실시간으로 모른다. 각자의 컨텍스트는 출발 시점에 고정되고, 그 이후 다른 에이전트가 만들어낸 변경을 인지하지 못한다. 이건 버그가 아니라 설계다. 그 설계를 이해하고 작업을 분해해야 한다. 컨텍스트 격리를 무시한 채 에이전트 수만 늘리면 속도가 아니라 혼란만 늘어난다.

## 발견한 적정 병렬도 — 3

이후 같은 규모의 작업을 병렬 3으로 줄여서 돌렸다. 두 가지가 달라졌다. 첫째, 에이전트 수가 적어지니 공유 파일 충돌 가능성이 낮아졌다. 둘째, 3개 작업이 완료된 뒤 남은 2개를 순차로 처리할 때, 먼저 완료된 결과물을 컨텍스트로 제공할 수 있었다.

wall-clock은 병렬 5보다 약 40% 더 걸렸지만 병합 후 수정 작업이 사라졌다. 전체 사이클 기준으로는 오히려 빨랐다. 병렬 5일 때 11분 완료 + 30분 병합 수정 = 41분. 병렬 3일 때 18분 완료 + 5분 검토 = 23분. 숫자로 놓고 보면 차이가 크다.

핵심은 병렬 실행 전에 작업 종속성 그래프를 먼저 그리는 것이다. "이 두 작업이 같은 파일에 손을 댈 가능성이 있는가"를 묻고, 가능성이 있으면 하나를 순차로 미룬다. 이 질문을 던지는 데 걸리는 시간은 5분이다. 그 5분을 아끼면 병합 단계에서 몇 배로 돌아온다.

```
# 의존성 없는 작업 3개를 한 메시지에 묶어 병렬 호출하는 패턴

task_a = Task(
    subagent_type="executor",
    prompt="엔드포인트 A 구현. 수정 대상 파일: src/routes/a.ts 전용."
)
task_b = Task(
    subagent_type="executor",
    prompt="엔드포인트 B 구현. 수정 대상 파일: src/routes/b.ts 전용."
)
task_c = Task(
    subagent_type="executor",
    prompt="엔드포인트 C 구현. 수정 대상 파일: src/routes/c.ts 전용."
)

# 세 Task를 단일 메시지에 담아 동시에 발화 — 순차 await 금지
results = await asyncio.gather(task_a, task_b, task_c)

# task_d, task_e는 results를 컨텍스트로 받아 순차 처리
task_d = Task(subagent_type="executor", prompt=f"엔드포인트 D. 기존 결과: {results[0]}")
```

> 병렬은 속도가 아니라 합의의 비용이다.

## 다음 실험에서 바꿀 것

충돌 이후 추가한 규칙이 있다. 각 에이전트 프롬프트에 "수정 가능한 파일 목록"을 명시하고, 그 목록 밖의 파일은 읽기만 하고 쓰지 말 것을 지시한다. 간단한 규칙인데 충돌이 사라졌다.

| 파일 구분 | 규칙 |
|---|---|
| 에이전트 전용 파일 (routes/a.ts) | 해당 에이전트만 쓰기 가능 |
| 공유 유틸리티 (utils/*.ts) | 읽기만. 수정 필요 시 오케스트레이터에 위임 |
| 공유 타입 정의 (types/*.ts) | 읽기만. 새 타입 추가도 오케스트레이터 통해서 |

오케스트레이터가 공유 파일 변경을 일괄 처리하면 충돌이 없어진다. 에이전트는 자기 영역 안에서만 논다.

5개를 동시에 띄워서 얻은 가장 중요한 교훈은 작업 분해가 병렬화보다 선행돼야 한다는 것이다. "독립적인 작업인가"를 확인하는 시간을 아끼면 병합 단계에서 그 이상을 쓴다. 3개 병렬 + 의존성 그래프 먼저 그리기 — 이걸 다음 프로젝트에 기본값으로 쓸 생각이다.

다음에 바꿀 것은 두 가지다. 하나는 에이전트 프롬프트에 "이 파일 외에는 쓰지 말 것" 한 줄을 반드시 넣는 것. 이번 사고에서 가장 단순하고 효과적인 방어였다. 다른 하나는 병렬 작업을 시작하기 전에 오케스트레이터가 공유 파일 목록을 먼저 확정하는 단계를 넣는 것이다. 작업마다 공유 파일이 뭔지 명시하고, 그 파일에 대한 변경은 전부 오케스트레이터를 통하게 만들면 충돌 자체가 구조적으로 불가능해진다. 이 두 가지는 다음 외주 프로젝트에서 바로 적용할 생각이다.

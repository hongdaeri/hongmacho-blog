아이디어 노트에는 늘 뭔가 적혀 있었다.

"독서 기록 + 간격 반복 복습을 합친 앱", "ETF 포트폴리오 드리프트를 시각화해주는 CLI", "Spring Boot 로그를 자연어로 설명해주는 도구". 생각날 때마다 메모해 두는데, 막상 만들지는 않는다. 주말에 Next.js 초기화하고 shadcn 세팅하고 DB 스키마 짜다 보면 저녁이 다 지나가 있다. 코어 기능 한 줄도 못 건드리고.

이게 반복되다 보니 어느 순간 "아이디어를 메모하는 습관 자체가 문제인 건 아닐까"라는 생각이 들었다. 메모하면 만든 것 같은 기분이 들어서 실제로는 안 만드는 것. 아이디어를 노트에 묻어두는 게 아니라, 그 자리에서 바로 프로토타입이 나오면 어떨까. `auto-project-builder`는 그 질문에서 시작했다.

## 왜 하필 Claude Code 스킬이었나

처음엔 직접 쓸 스크립트를 만들려 했다. Next.js 보일러플레이트를 자동 생성하는 shell script 수준으로. 그런데 그건 이미 있다. `create-next-app`, `create-t3-app`, 템플릿은 수백 개다. 내가 매번 막히는 건 세팅이 아니었다.

"이 아이디어, 경쟁 서비스랑 뭐가 다른가", "어떤 기능부터 만들어야 하나", "이게 실제로 쓸 만한 물건인가"를 판단하는 데 에너지가 더 들었다. 코드를 한 줄 쓰기 전에 이미 지쳐있는 경우가 많았다. 그러니까 내가 자동화하고 싶었던 건 코드 생성이 아니라 판단 과정이었다. 아이디어를 평가하고, 기획하고, 우선순위를 정하는 것.

그게 Claude Code 스킬 형태로 만들면 자연스럽다고 판단했다. 에이전트들이 각자의 역할로 판단을 나눠서 하는 구조이니까. `auto-blog-write`를 먼저 만들면서 패턴을 익혔던 것도 이유였다. 주제 하나를 넣으면 카테고리 분류, 슬러그 생성, 글 작성, 빌드 검증, 커밋·푸시까지 자동으로 끝나는 스킬. 그게 잘 돌아가는 걸 보고, 같은 구조로 프로젝트 빌드도 될 것 같다는 확신이 생겼다. 블로그 글 한 편보다 복잡하지만, 기본 형태는 같다. 입력 → 평가 → 생성 → 검증 → 배포.

스킬 명세를 처음 쓰기 시작했을 때는 200줄을 넘기면 안 된다고 스스로 선을 그었다. 단순해야 잘 돌아간다고 생각했다. 결과적으로 명세는 1,400줄이 됐다. 트렌드 조사, 아이디어 평가, YC 스타일 스코어카드, OMC/ECC/none 환경 분기, 자동 QA 루프, 체크포인트 재개, 보고서 HTML 생성, 고도화 제안 페이지까지 하나씩 추가하다 보니 그렇게 됐다. 단순함을 지키려는 의지와, 엣지 케이스를 처리하려는 의지가 충돌하면 항상 후자가 이긴다. 그리고 어느 시점이 되면 그게 당연해진다. 200줄이 충분했던 시점이 언제였는지도 기억이 안 날 만큼.

명세를 쓰면서 가장 오래 걸린 부분은 OMC_MODE 분기였다. 로컬 환경인지, ECC(Everything-Claude-Code) 환경인지, 아무것도 없는 기본 환경인지에 따라 호출하는 에이전트가 달라진다. 로컬에선 내가 직접 설치한 에이전트 체인을 쓰고, ECC에선 플러그인 형태의 에이전트를 쓰고, none일 땐 Claude Code 기본 기능만으로 돌아가야 한다. 세 가지를 다 처리하면서 핵심 파이프라인을 유지하는 게 생각보다 복잡했다.

## 처음 실행한 날

스킬 명세를 완성하고 처음 실행한 건 2026년 4월 말, 저녁 11시쯤이었다.

설정 질문에 웹(Web), Next.js, 개발자 도구 카테고리, 3개를 입력했다. 그 뒤로는 손대지 않았다. 트렌드 조사가 돌고, 아이디어 평가가 돌고, PRD가 써지고, Sprint 계획이 잡히고, 코드가 생성됐다. 나는 그냥 다른 탭에서 책을 읽었다.

두 시간쯤 지나 알림이 떴다. macOS 알림 — "git-diff-explainer 빌드 완료 (1/3)". 처음엔 뭔가 잘못됐다고 생각했다. 실제로 빌드가 됐다는 게 실감이 안 났다.

프로젝트 폴더를 열어봤다. `git-diff-explainer`, `api-mock-studio`, `env-doctor`. 세 개 전부 GitHub에 올라가 있었다. README도 있고, 테스트도 있고, `npm run dev`를 치니까 실제로 뭔가 뜬다. 코드를 훑어보니 내가 짠 것만큼 꼼꼼하지는 않았지만, 돌아간다. 컴포넌트가 좀 크고, 에러 처리가 일부 빠져 있고, 변수명이 간혹 generic했다. 하지만 QA를 통과했고, 핵심 기능은 동작했다.

2시간 만에 GitHub 저장소 3개가 생겼다. 내가 혼자 만들었으면 초기 세팅과 설계에만 주말 하루를 썼을 분량이다. 아이디어 하나를 실제로 돌아가는 물건으로 만드는 데 걸리는 시간이 이렇게 줄어들 수 있다는 게, 처음엔 조금 당황스러웠다. 기쁜 게 아니라 당황스러웠다는 게 정확하다. 뭔가 내가 한 게 없는 기분이랄까. 만들었는데 만든 느낌이 별로 없는 이상한 감각.

그래도 한 가지는 분명했다. 이전에는 아이디어가 노트에서 머물렀다면, 이제는 아이디어가 레포로 이동한다. 노트에 적힌 것들이 언제가 되어도 코드가 안 됐는데, 이건 2시간 안에 코드가 됐다. 그 차이가 작지 않다.

## 기대와 달랐던 것들

결과가 나왔을 때 예상한 감정은 "와, 편하다"였다. 실제로 든 감정은 좀 달랐다.

먼저 아이디어의 독창성 문제다. 스킬이 선정한 아이디어들이 대부분 이미 있는 카테고리의 변형이었다. `git-diff-explainer`는 비슷한 도구가 이미 여럿 있고, `api-mock-studio`도 Mockoon 같은 기존 솔루션이 있다. 15점 YC 스코어카드로 평가하는데, 그 평가 자체가 기존 정보에 기반하니 이미 포화된 시장을 향하는 경향이 생겼다.

Reddit과 HN에서 많이 언급되는 고통은 이미 많은 사람이 해결하려 시도한 고통이기도 하다. 미충족 수요를 찾는 방법론인데, 찾기 쉬운 미충족 수요는 이미 충족하려는 시도가 많다는 뜻이기도 하다. 이건 스킬의 한계라기보다 트렌드 분석 방법 자체의 한계에 가깝다. 독창적인 아이디어는 기존 데이터에서 나오기 어렵다. 잘 알려지지 않은 고통일수록 검색에 안 걸린다.

코드 품질은 QA를 통과했지만 "동작함"과 "쓸 만함" 사이에 간극이 있었다. `api-mock-studio`를 직접 써보니 버튼 위치가 직관적이지 않고, 에러 메시지가 너무 기술적이었다. 기능은 다 있는데 UX가 어색한 물건. 빌드 통과는 동작을 보장하지 사용성을 보장하지 않는다는 걸 실감했다. QA 루프가 기능 테스트 위주라서, 실제 사용자 흐름에서 발생하는 어색함을 잡지 못한다. 이건 QA 기준을 높이거나 Playwright 기반 UX 체크를 추가하면 개선할 수 있다고 생각한다.

| 항목 | 예상 | 실제 |
|------|------|------|
| 아이디어 독창성 | 새로운 분야 발굴 | 기존 카테고리 변형 |
| 코드 품질 | 최소 동작 수준 | 동작하지만 UX 어색함 |
| PRD 품질 | 대략적인 수준 | 예상보다 구체적 |
| 소요 시간 | 4~5시간 예상 | 2시간 |

생각보다 좋았던 건 PRD 품질이었다. planner 에이전트가 작성한 PRD에는 타겟 페르소나가 구체적으로 잡혀 있었고, 경쟁 서비스 분석도 읽을 만했다. 내가 직접 PRD를 쓸 때보다 구조가 더 체계적인 경우가 있었다. 시장 크기 추정, 핵심 사용자 시나리오, 2주 Sprint 계획까지 포함된 문서가 20분 안에 나온다. 이건 실제로 유용하다는 확신이 생겼다.

## 동기가 바뀌었다

처음 동기는 "아이디어를 빠르게 프로토타입으로 만들고 싶다"였다.

스킬을 완성하고 몇 차례 돌려보고 나서 동기가 다르게 정리됐다. 이 스킬이 잘하는 게 "아이디어를 프로토타입으로 만드는 것"이라기보다 "아이디어를 제대로 된 형태로 구체화하는 것"에 더 가깝다는 걸 알았다. PRD, 경쟁 분석, Sprint 계획을 빠르게 뽑아주는 것. 그 과정에서 "이건 별로다"라는 판단도 함께 나온다.

실제로 유용하게 쓴 경우는 내가 이미 만들고 싶은 게 있을 때 아이디어를 직접 입력(`USER_IDEA` 경로)해서 PRD와 초기 구현을 뽑는 방식이었다. 완전 자율 모드보다 이게 더 실용적이었다. 스킬이 알아서 아이디어를 선정하면 내가 별로 관심 없는 카테고리가 나올 수 있는데, 내가 관심 있는 주제를 넣으면 그 주제에 집중한 결과가 나온다. "자율"보다 "반자율"이 더 잘 맞았다.

메모만 하고 안 만들던 습관이 완전히 바뀌진 않았다. 다만 아이디어를 노트에 적는 대신 스킬에 입력해서 PRD까지 빠르게 만들어 두는 걸로 바꼈다. 만들지는 않더라도, 구체적인 형태로 남겨 두는 것. 나중에 진짜 만들고 싶을 때 다시 꺼내면 출발점이 이미 있다. "언젠간 만들어야지"라는 말이 "이미 PRD가 있는데"로 바뀐 것만으로도 심리적 마찰이 줄어든다. 그게 작아 보여도 실제로 행동에 영향을 준다.

한 가지 더. 스킬을 만드는 과정 자체가 내가 무엇을 자동화하고 싶은지를 명확하게 만들어줬다. "아이디어를 빨리 만들고 싶다"가 실제로는 "판단 과정의 마찰을 줄이고 싶다"라는 걸, 1,400줄짜리 명세를 쓰면서 알았다. 코드보다 명세를 쓰는 과정이 더 유익했다는 게 조금 아이러니하다.

## 아직 검증 못 한 가정들

스킬을 만들면서 검증하지 못한 가정이 몇 가지 있다.

가장 큰 것은 "자동으로 생성된 프로젝트를 실제로 사용하는 사람이 있을 것인가"다. GitHub에 올라가 있고 Star를 받았다면 효용이 증명된다. 지금은 프로젝트가 올라가 있는 것까지만 확인했고, 그 이후 사용 데이터가 없다. 몇 달 트래킹해 봐야 안다. 생성된 코드가 단순히 "돌아가는 데모" 수준인지, 누군가 실제 업무에 쓸 수 있는 수준인지는 Star 수보다 실사용 패턴을 봐야 알 수 있다.

두 번째는 비용이다. 프로젝트 3개에 트렌드 조사, idea-generator, PRD 작성, 코드 생성, QA 루프를 전부 돌리면 토큰이 상당히 나간다. 아직 정확히 계산하지 않았다. 2시간이라는 시간 절약 대비 토큰 비용이 합리적인지 따져보는 게 남아 있다. 시간당 인건비로 환산하면 이득인지, 아니면 단순한 편의 비용인지. 이 계산이 되어야 "이 스킬을 정기적으로 쓸 이유가 있다"는 결론을 낼 수 있다.

세 번째는 품질의 천장이다. 지금은 QA 통과가 기준인데, 더 높은 기준 — 실제 사용자 피드백, 코드 유지보수성, 보안 취약점 없음 — 을 넣으면 어디서 무너지는지 아직 모른다. 기준을 높이면 QA 루프 재시도 횟수가 늘어나고, 그러면 비용과 시간이 올라가고, 그 균형점이 어디인지 실험해 봐야 한다.

이 세 가지를 확인하는 게 다음 단계다. 프로젝트들의 GitHub 활동을 몇 달간 트래킹하고, 토큰 비용을 산출하고, 더 높은 품질 기준을 붙여서 돌려보는 것. 지금은 일단 돌아가는 물건을 만든 단계까지고, 그게 실제로 가치 있는지는 시간이 알려줄 것이다.

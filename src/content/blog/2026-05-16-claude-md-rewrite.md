지난달 외주 프로젝트에서 Claude Code를 본격적으로 쓰기 시작했을 때, CLAUDE.md 파일은 세 줄이었다. 기술 스택, 빌드 명령, 테스트 실행 방법. 에이전트가 코드는 짰는데 PR 설명이 매번 달랐다. 어떤 PR은 영어로, 어떤 PR은 한국어로. 어떤 커밋은 간결하고 어떤 커밋은 장황했다. 도구를 쓰는 건데 결과가 일관성 없이 튀었다. 처음엔 에이전트 품질 문제라고 생각했다. 프롬프트를 더 구체적으로 써보기도 하고, 예시를 직접 붙여주기도 했다. 그 방법이 해당 세션에서는 먹혔지만 다음 세션엔 다시 원점이었다.

문제가 CLAUDE.md에 있다는 걸 깨닫는 데 이틀이 걸렸다. 에이전트는 내가 알려준 것만 안다. 나는 당연하게 여기는 것들을 하나도 알려주지 않은 채 "이 스타일대로 해줘"를 기대하고 있었다.

## 다시 쓰기 전 한 주 동안 무슨 일이 있었나

결정적 사례가 하나 있었다. 같은 태스크를 세 번 PR로 받았는데 셋 다 달랐다. 첫 번째 PR은 함수명이 `getUserInfo`였고, 두 번째는 `fetchUser`였고, 세 번째는 `getUser`였다. 같은 에이전트, 같은 프롬프트, 같은 맥락인데 결과물이 달랐다. 커밋 메시지 종결체도 첫 번째는 "~합니다", 두 번째는 "~함", 세 번째는 영어였다. 코드 내 주석은 어떤 건 없고, 어떤 건 한 줄짜리 한국어, 어떤 건 두 줄짜리 영어였다.

리뷰 때마다 같은 피드백을 반복했다. "함수명 일관성 맞춰주세요", "커밋 메시지 형식 지켜주세요". 그 피드백이 다음 PR에 반영되지 않은 건 에이전트가 게으른 게 아니었다. 내가 맥락을 저장해두지 않았기 때문이다. 각 세션이 끝나면 그 피드백은 사라졌다. CLAUDE.md만이 세션을 넘어 남는 기억이었는데, 거기엔 아무것도 없었다.

## 기존 CLAUDE.md의 문제

```markdown
## Tech Stack
- Next.js 15, TypeScript, Tailwind CSS
- Prisma + PostgreSQL
- Vitest for testing

## Build
npm run build

## Test
npm run test
```

이게 전부였다. 에이전트 입장에서 이 파일은 실행 환경 안내서다. 톤도 없고, 금기도 없고, 디렉터리 맥락도 없다. PR을 어떤 형식으로 써야 하는지, 커밋 메시지는 어떤 규칙인지, 어떤 패턴을 쓰면 안 되는지 — 아무것도 없었다.

결과는 예측 가능했다. 에이전트는 훈련 데이터에서 가장 흔하게 본 패턴을 꺼냈다. 영어 PR 설명, 장황한 주석, "TODO: implement later" 같은 placeholder — 전형적인 AI-slop 패턴이 1.4건/PR 꼴로 나왔다.

더 당황스러운 건 에이전트를 탓할 수가 없다는 거였다. 에이전트 입장에서는 최선을 다한 결과물이었다. 훈련 데이터에서 본 가장 일반적인 코드 스타일, 가장 일반적인 커밋 메시지 형식을 따랐을 뿐이다. 내가 원하는 것이 그 일반적인 것과 다르다면, 그 차이를 나 스스로 명시해야 했다. 프롬프트로 매번 설명하는 건 한계가 있다. 세션이 끝나면 사라지니까. 파일로 남겨야 했다.

## 다시 쓴 6개 섹션

```markdown
## 1. Tone & Style
- 한국어로 답한다. PR 설명, 커밋 메시지, 인라인 주석 모두 한국어.
- 커밋 메시지: type(scope): 한국어 설명 (50자 이내)
- 주석은 "왜"만 적는다. "무엇"은 코드가 말한다.

## 2. Forbidden Patterns
- TODO / FIXME / HACK 주석 금지. 해결하거나 이슈로 분리한다.
- console.log 디버그 코드 커밋 금지.
- any 타입 금지. unknown 또는 구체 타입 사용.
- placeholder 함수(throw new Error("not implemented")) 금지.

## 3. Directory Map
src/
  app/          # Next.js App Router — 라우팅만
  components/   # UI 컴포넌트 — 비즈니스 로직 없음
  lib/          # 순수 유틸리티 — 외부 의존 없음
  server/       # 서버 액션 및 DB 쿼리
  types/        # 공유 타입 정의

## 4. Common Commands
npm run dev          # 개발 서버 (포트 3000)
npm run build        # 프로덕션 빌드
npm run test         # Vitest 유닛 테스트
npm run lint         # ESLint + Prettier 검사
npm run typecheck    # tsc --noEmit

## 5. Build Verification
커밋 전: typecheck → lint → test 순서로 전부 통과해야 한다.
빌드 에러가 있으면 커밋하지 않는다. 경고도 0이 목표다.

## 6. PR Template
## 변경 내용
- 항목 형식으로 구체적으로

## 테스트 방법
- 검증한 시나리오 명시

## 관련 이슈
- #이슈번호 (없으면 생략)
```

총 40줄. 기존의 8배 길이다. 길어진 게 문제가 아니라, 에이전트가 판단해야 하는 여지를 줄인 게 핵심이다. 에이전트가 스스로 결정을 내릴 때마다 일관성이 깨진다. 결정을 미리 내려서 파일에 박아두면, 에이전트는 실행에만 집중한다. 그게 CLAUDE.md의 역할이다.

## 어떤 항목을 넣을지 정한 기준

6개 섹션을 구성할 때 "좋은 규칙"을 나열하지 않으려 했다. 그 기준이 되면 끝없이 추가할 수 있다. 대신 한 가지 질문으로 필터링했다. "에이전트가 이 프로젝트에서 가장 자주 틀리는 부분이 뭔가."

톤 규칙이 첫 번째로 들어간 건 그래서다. 커밋 메시지 언어, 주석 언어, PR 설명 형식 — 이 셋이 매 세션마다 제각각이었다. Forbidden Patterns는 두 번 이상 반복해서 내가 직접 지워야 했던 패턴들만 넣었다. `TODO` 주석은 세 번, `console.log`는 다섯 번 커밋 직전에 내가 잡아냈다. Directory Map은 에이전트가 서버 로직을 `components/`에 넣었던 날 추가했다. 틀릴 때마다 섹션이 생겼다.

PR Template은 유일하게 선제적으로 넣은 항목이다. 아직 에이전트가 틀리기 전에 넣었다. 이전 프로젝트에서 PR 설명이 없는 채로 머지됐다가 3주 뒤 히스토리를 추적할 수 없었던 경험이 있어서다. 같은 실수를 이 프로젝트에서 반복하기 싫었다.

## 1주 전후 비교

섹션을 추가한 다음 주에 같은 태스크 3개를 돌렸다. 전주와 동일한 규모의 기능 추가 태스크였다.

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 평균 PR 설명 길이 | +28% 장황 | 필요한 내용만 |
| AI-slop 잡음 표현 | 1.4건 / PR | 0건 |
| 변경 코드 라인 수 | 기준값 | -18% |
| 영어 혼용 | 매 PR 발생 | 0건 |

코드 라인이 줄어든 건 흥미로웠다. forbidden patterns 섹션에서 placeholder와 TODO를 막으니 에이전트가 처음부터 완성된 코드를 짰다. 미완성 코드를 추가하는 대신 범위를 줄여서 완성도를 택했다. 이건 의도한 효과였는데 실제로 작동했다.

수치로 가장 체감이 컸던 건 PR 리뷰 시간이었다. CLAUDE.md를 바꾸기 전 일주일 동안 PR 하나를 리뷰하는 데 평균 18분이 걸렸다. 잡음을 걷어내는 데 절반 이상이 쓰였다. 언어를 통일하고, 불필요한 주석을 지우고, 변수명 일관성을 맞추는 작업이었다. 바꾼 뒤에는 평균 10분으로 줄었다. 코드 자체를 보는 시간이 같아도 주변 정리 작업이 사라지니까 집중이 달랐다. 일주일에 PR을 6-8개 처리하면, 절약되는 시간이 누적된다.

> 에이전트는 CLAUDE.md의 크기만큼 신중해진다.

## 계속 고쳐나가는 문서

CLAUDE.md는 한 번 쓰고 끝나는 파일이 아니다. 에이전트가 반복적으로 틀리는 패턴이 보이면 그때 섹션을 추가한다. 이번 주엔 "서버 컴포넌트와 클라이언트 컴포넌트 판단 기준" 섹션을 하나 더 넣었다. 에이전트가 `use client`를 습관적으로 붙이는 경향이 있어서다. 인터랙션이 전혀 없는 컴포넌트에도 붙이길래, 판단 기준을 명문화했더니 그 다음 PR부터 사라졌다.

운영 방식은 간단하다. 에이전트 출력을 리뷰하다 "이건 두 번째 본다"는 패턴이 나오면 즉시 CLAUDE.md에 추가한다. 쌓이다 보면 이 파일이 프로젝트의 암묵지 문서가 된다. 팀에 새 사람이 들어왔을 때 온보딩 문서로 줘도 될 만큼 밀도가 생긴다.

주의할 점이 하나 있다. CLAUDE.md가 너무 길어지면 에이전트가 중간 섹션을 건너뛰기 시작한다. 긴 문서는 앞부분과 뒷부분이 우선순위가 높고 중간이 흐려진다. 경험상 150줄을 넘어가면 맨 아래 섹션 준수율이 떨어졌다. 그래서 분기마다 한 번씩 섹션을 정리한다. 더 이상 문제가 안 되는 패턴은 삭제한다. 살아있는 문서는 추가만 하는 게 아니라 덜어내기도 한다.

기술 스택만 적어놓는 CLAUDE.md는 사용 설명서다. 규칙과 맥락까지 적어놓은 CLAUDE.md는 팀의 코딩 규범이다. 에이전트는 그 차이를 정확히 안다.

지난주 월요일 오전, Astro 6 마이그레이션 작업을 시작하다가 Claude Code에 Context7을 붙였다. 붙이고 나서 한 시간도 안 돼 "이걸 왜 이제 했지"라는 생각이 들었다. 그게 시작이었다. 그 주에 MCP 서버를 두 개 더 붙였고, 첫 주를 보낸 뒤 뭐가 달라졌는지 정리해본다. 설정 과정에서 삽질도 있었고, 예상치 못한 단점도 있었다.

## MCP가 뭔지 먼저 — 짧게

MCP(Model Context Protocol)는 Claude Code가 외부 도구·데이터 소스와 표준화된 방식으로 통신할 수 있게 해주는 프로토콜이다. 서버 하나를 붙이면 Claude Code가 그 서버의 기능을 도구처럼 호출할 수 있다. 브라우저 자동화, 문서 검색, 외부 API 호출 같은 작업을 Claude Code 안에서 끊김 없이 처리할 수 있다는 뜻이다.

이번 주에 붙인 서버는 세 개다.

- **Context7**: 라이브러리·프레임워크 공식 문서를 실시간으로 가져온다.
- **Playwright**: 브라우저를 직접 조작한다. 스크린샷, 클릭, 폼 입력.
- **sequential-thinking**: 긴 추론 체인이 필요한 작업에 단계별 사고 구조를 붙여준다.

## Context7 — 가장 즉각적인 변화

Astro 6으로 마이그레이션하면서 가장 자주 마주친 문제는 "API가 바뀌었는데 정확히 어떻게 바뀌었지?"였다. 예전 방식은 이랬다: 구글에서 검색하고, 공식 문서 탭을 열고, 원하는 섹션을 찾고, Claude Code 창으로 돌아와서 붙여넣고 다시 설명을 요청하는 과정을 반복했다.

Context7이 붙고 나서 이 루프가 사라졌다. Claude Code 안에서 "Astro 6의 Content Layer API 사용법 알려줘"라고 입력하면 Context7이 공식 문서에서 관련 섹션을 바로 가져온다. 탭 전환이 없다.

첫날 마이그레이션 작업 시간이 평소의 70% 수준으로 줄었다. 정확한 측정은 아니지만 체감이 뚜렷했다. 특히 효과가 컸던 건 "이 API가 버전 X에서 어떻게 바뀌었지?"라는 질문 유형이었다. 이런 질문은 구글 검색에서 잘 걸리지 않는다. 공식 마이그레이션 가이드 문서 깊숙이 들어가야 나오는 내용인데, Context7이 그 구간을 정확하게 끊어서 가져왔다. LLM의 학습 데이터에 없는 최신 API 변경사항을 실시간으로 가져온다는 점에서, 단순한 편의 도구가 아니라 정확도를 높이는 도구에 가깝다는 인상을 받았다.

## 설정 트러블슈팅 — 사소한 실수들

설정 과정이 매끄럽지는 않았다. Context7을 붙이는 데만 첫날 30분 가까이 소비했다.

처음엔 Context7 공식 README에 나온 MCP 서버 URL을 그냥 복사해서 붙여넣었다. 그런데 URL 형식이 두 가지였다 — npm 패키지 경로와 GitHub 저장소 URL이 문서 곳곳에 섞여 있었다. 어느 게 실제로 써야 하는 형태인지 구분이 안 됐다. 처음엔 GitHub URL을 `command` 필드에 넣었다가 서버가 뜨지 않았고, 두 번째로 npm 패키지 이름을 넣었는데 이번엔 `args` 배열 형식이 틀렸다는 오류가 났다. 결국 GitHub README의 raw JSON 설정 블록을 그대로 복사해서 붙여넣는 방식으로 해결했다. 문서를 읽고 이해하는 것보다 예제 JSON을 통째로 가져오는 게 더 빠른 경우가 있다.

**경로 혼동**: `.claude/mcp.json` 경로를 `~/.claude/mcp.json` 전역 설정 파일로 헷갈렸다. 전역 설정에 쓰면 모든 프로젝트에 서버가 붙는다. 프로젝트별로 켜고 끄고 싶다면 프로젝트 루트의 `.claude/mcp.json`을 써야 한다. 당연한 구분인데 처음엔 파일이 어디에 있어야 하는지 확인하지 않고 그냥 썼다.

**따옴표 이슈**: NPX 커맨드에서 큰따옴표가 누락됐다. Context7 서버 등록 시 `args` 배열 안에 패키지 이름을 넣는데, JSON 안에서 문자열을 작은따옴표로 쓰면 파싱 오류가 난다. JSON은 무조건 큰따옴표다. 당연한 규칙인데 터미널에서 에코로 확인하지 않고 바로 저장했다가 서버가 뜨지 않았다.

아래는 Context7을 등록하는 `.claude/mcp.json` 기본 형태다.

```json
{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp@latest"],
      "env": {}
    }
  }
}
```

Playwright와 sequential-thinking도 같은 구조로 추가하면 된다. 서버마다 `command`와 `args`만 바뀐다.

## Playwright — 예상보다 쓸 일이 많다

브라우저 자동화가 에이전트 작업에서 이렇게 자주 필요할 줄 몰랐다. 첫 주에 쓴 사례만 세 가지다.

1. 배포 후 특정 페이지가 실제로 렌더링되는지 스크린샷으로 확인.
2. 블로그 OG 태그가 SNS 미리보기에서 제대로 보이는지 검증.
3. 로컬 개발 서버에서 네비게이션 링크 클릭 동작 테스트.

전부 "코드를 짜서 검증하는 것"보다 "Claude Code가 브라우저를 열어서 직접 보는 것"이 훨씬 빠른 작업들이었다.

## 단점 — 활성 도구는 좁게 유지해야 한다

첫 주에 느낀 가장 큰 단점은 토큰 소모다. MCP 서버가 붙으면 Claude Code가 세션을 시작할 때 각 서버의 도구 목록과 스키마를 컨텍스트에 로드한다. 서버 하나당 수백에서 수천 토큰이 초기에 소모된다. 세 개가 동시에 붙으면 짧은 작업에서도 입력 토큰이 눈에 띄게 늘어난다.

해결책은 간단하다. 프로젝트별 설정을 활용해서 그 프로젝트에서 실제로 쓸 서버만 켜두는 것이다. Playwright는 프론트엔드 프로젝트에만, sequential-thinking은 복잡한 설계 작업에만. Context7은 새로운 라이브러리를 쓸 때만 켠다. 도구가 많을수록 고를 게 많아 보이지만, 실제로는 어떤 도구가 언제 쓰이는지 불분명해져서 에이전트 판단이 흐려진다.

토큰 소모 외에 한 가지 더 있었다. 서버 세 개가 동시에 붙어 있으면 Claude Code가 간단한 질문에도 MCP 도구 호출을 시도하는 경향이 생겼다. "이 변수 이름 뭐가 좋을까"처럼 문서 조회가 전혀 필요 없는 질문에도 Context7을 호출하려는 시도가 간간이 끼어들었다. 도구 목록이 길면 에이전트가 도구를 더 적극적으로 쓰려 한다는 부작용이다. 활성 서버를 작업 목적에 맞게 좁히는 것이 응답 품질을 유지하는 데도 도움이 됐다.

> 도구의 폭은 늘리되, 활성 도구는 좁게.

## 한 주가 지나고

세 서버를 붙인 뒤 작업 패턴이 바뀌었다. 가장 큰 변화는 "컨텍스트 전환 횟수"가 줄었다는 거다. 탭을 왔다 갔다 하거나, 터미널을 따로 열거나, 캡처한 문서를 붙여넣는 행위가 줄었다. Claude Code 안에서 끝낼 수 있는 작업의 범위가 넓어졌다.

체감상 가장 달라진 건 흐름이 끊기지 않는다는 점이다. 예전엔 구현 중에 API 한 줄이 헷갈리면 브라우저로 넘어가서 검색하고, 다시 에디터로 돌아오는 과정에서 집중이 흩어졌다. 그 전환이 하루에 수십 번 일어난다. MCP가 그 이탈을 Claude Code 안에서 흡수해주면서 하나의 작업을 끝까지 붙잡고 있는 시간이 늘었다. 생산성 도구의 가치는 기능의 수가 아니라 끊김의 수를 줄이는 데 있다는 걸 다시 확인했다.

다음 주엔 GitHub MCP 서버를 붙여볼 생각이다. PR 생성·리뷰 코멘트 읽기·이슈 트래킹을 Claude Code 안에서 처리하면 외주 프로젝트 관리 흐름이 더 단순해질 것 같다. 붙이고 나면 또 정리해둘 예정이다. 도구는 써봐야 한계가 보인다.

블로그 글을 쓰는 데 가장 오래 걸리는 구간은 첫 줄이었다. 주제는 있는데 어떻게 시작할지 몰라서 빈 에디터를 20분씩 쳐다봤다. 어느 날 그 시간이 아깝다는 생각이 들었고, Claude Code에 `.claude/commands/blog-auto-write.md`를 만들었다. 그 이후로 첫 줄까지 걸리는 시간이 30분에서 5분으로 줄었다.

## 자동화 전에 어떻게 썼나

자동화를 만들기 전에 블로그 초안이 어떻게 죽었는지를 먼저 적는다. 그래야 왜 이 구조가 나왔는지가 납득된다.

2025년 한 해 동안 완성된 글은 11편이었다. 같은 기간 드래프트 폴더에 쌓인 미완성 파일은 23개였다. 절반이 넘는 글이 초안 상태로 묻혔다. 이유는 거의 항상 같았다. 주제는 있는데, 첫 문단을 어떤 톤으로 시작할지 결정하지 못한 채 30분이 지나면 창을 닫았다. 다음에 다시 열었을 때는 처음의 의욕이 식어 있었다.

더 구체적으로 보면, 초안 파일에는 세 가지 패턴이 있었다. 첫째, 제목만 있고 본문이 없는 파일. 주제를 메모해두고 "나중에 쓰자"로 남긴 것들이다. 둘째, 첫 문단만 쓰다 멈춘 파일. 도입부를 세 번쯤 고쳐 쓰고 포기한 흔적이 있다. 셋째, 절반쯤 쓰다 막힌 파일. 중간 섹션에서 "이 흐름이 맞나" 싶어서 구조를 다시 짜다가 방치된 것들이다. 세 번째 유형이 가장 아깝다. 글의 덩어리는 있는데, 구조 결정을 못 해서 죽은 거니까.

30분짜리 "어떤 톤으로 시작하지" 막힘은 생각보다 일관된 패턴이었다. 1인칭 단언으로 시작할지, 상황 묘사로 시작할지, 숫자로 시작할지. 그 선택을 매번 처음부터 했다. 같은 선택을 스물세 번 반복하다가 스물세 번 다 시간이 지나서 포기했다. 그 판단 자체를 명령어 파일에 위임하는 게 자동화의 핵심이었다. 톤과 구조 결정을 프롬프트 파일에 박아두면, 내가 매번 처음부터 선택할 필요가 없어진다.

드래프트 폴더를 보다가 또 하나를 발견했다. 절반 이상이 완성된 글도, 마지막 섹션을 쓰다가 "이 글이 어디서 끝나야 하지"를 못 정해서 방치된 것들이었다. 결말을 어떻게 닫을지 모르는 상태에서 글이 열려 있으면 손이 안 간다. 명령어 파일에 섹션 구조와 마감 방식까지 정의해두면 그 망설임도 사라진다. 지금 명령어 파일에 "마지막 섹션은 미결 과제나 다음 실험으로 닫는다"는 지침이 있는 이유가 거기 있다.

결국 자동화가 겨냥한 건 특정 작업이 아니라 특정 순간이었다. 주제를 잡고 나서 첫 문장이 나오기까지의 공백, 그 공백에서 포기가 일어났다. 그 간격을 없애는 게 목표였고, 명령어 파일은 그 수단이었다. 23개의 미완성 파일이 그 결정을 내리게 한 증거였다. 숫자를 세고 나서야 패턴이 보였다.

## 슬래시 명령어가 뭔지 먼저

Claude Code는 `.claude/commands/` 디렉토리에 마크다운 파일을 두면 `/파일명`으로 호출할 수 있다. 파일 안에 프롬프트를 쓰면 된다. 특수한 환경 없이 레포에 파일 하나 추가하면 끝이라 진입 비용이 거의 없다.

`/blog-auto-write 주제`를 치면 Claude Code가 그 파일의 프롬프트를 실행한다. 나는 거기에 톤 규칙, 스키마 검증, 빌드 명령까지 전부 넣었다. 프롬프트 파일이 길어질수록 결과물이 안정적이었다. 짧게 "블로그 글 써줘" 한 줄보다, 금기어와 스키마와 실행 순서를 명시한 100줄 파일이 훨씬 일관된 결과를 냈다.

## 명령어 파일에 무엇을 넣었나

핵심은 세 덩어리다: 금기어 목록, 프론트매터 스펙, 실행 절차.

```markdown
## FRONTMATTER SPEC
- date: YYYY-MM-DD (today's date)
- category: dev | invest | learn | daily
- excerpt: 140자 이하, 메타 설명 문체 금지
- readTime: 본문 한국어 글자 수 ÷ 600, 올림

## FORBIDDEN PHRASES (body에 절대 등장 금지)
- AI-slop 인사말 패턴 ("Hello everyone" 류 한국어 변형)
- 날짜 지시어 오프닝 ("Today we will..." 류)
- "~에 대해 살펴보겠습니다" 계열 설명 투
- 요약형 단락 오프닝 ("In conclusion..." 류)
- 에필로그 섹션 헤더 ("맺음말" 류)

## EXECUTION ORDER
1. Write .md to src/content/blog/YYYY-MM-DD-{slug}.md
2. Run: npm run build
3. If build passes, git add + commit
```

금기어 목록을 명령어 파일에 명시적으로 박아넣은 이유는, 제약 없이 생성하면 반드시 저 패턴 중 하나가 나오기 때문이다. 요약형 단락으로 시작하는 마지막 섹션이 등장하는 순간 글 전체의 신뢰가 흔들린다. 금기어 목록은 사후 체크리스트가 아니라 생성 자체를 막는 하드 필터로 운영했다.

excerpt 제약도 여기서 처리한다. "메타 설명 문체 금지"는 "~에 대해 알아봅니다" 같은 중립적 요약 투를 막기 위한 것이다. excerpt는 읽고 싶어지는 한 문장이어야 한다. 그 기준을 파일에 써두면 생성 결과물이 훨씬 달라진다.

## 스키마 검증과 빌드 연결

Astro는 `content/config.ts`에 zod 스키마가 있어서 프론트매터가 틀리면 `npm run build`가 실패한다. 이걸 검증 단계로 그대로 활용했다. 명령어 파일에 `npm run build`를 3단계에 넣어두면, Claude Code가 직접 빌드를 돌리고 실패 로그를 읽어서 프론트매터를 수정한다.

실제로 `excerpt`가 140자를 넘겼을 때 빌드가 한 번 실패했다. Claude Code가 에러 메시지를 읽고 excerpt를 줄여서 다시 빌드했다. 사람이 개입하지 않았다. 이 루프가 동작하는 걸 처음 봤을 때 생각보다 많이 신기했다.

```bash
# 명령어 실행 로그 (2026-05-04 실제)
$ /blog-auto-write "국내 ETF 6개월 후기"
→ 파일 생성: src/content/blog/2026-05-03-kr-etf-six-month.md
→ npm run build ... failed (excerpt 148자)
→ excerpt 수정 후 재빌드
→ npm run build ... passed
→ git commit -m "post: 국내 ETF 3종으로 6개월 굴려보고 정리한 것들"
```

빌드를 검증 게이트로 쓰는 방식은 다른 Astro 프로젝트에도 그대로 쓸 수 있다. zod 스키마가 있는 콘텐츠 레이어라면 어디서든 같은 패턴이 작동한다.

## 한 주 쓰고 나서

명령어를 만들고 일주일 동안 글 세 편을 올렸다. 이전 한 달 동안 두 편 올린 것과 비교하면 빈도가 의미 있게 올라갔다. 이유는 단순했다. 빈 에디터 앞에서 멈추는 시간이 없어졌다. 주제를 잡는 순간 첫 초안이 나와 있으니까, 내가 할 일은 그걸 읽고 수정하는 것만 남는다.

글을 처음부터 직접 쓰는 것과 초안을 수정하는 건 심리적으로 완전히 다른 작업이다. 전자는 빈 화면을 채워야 한다는 압박이 있고, 후자는 이미 있는 것을 고친다는 감각이다. 그 차이가 진입 비용을 바꿨다. 수정하는 데 15분이 걸렸다. 처음부터 쓰는 데는 45분이 걸렸다. 같은 글이지만 체감 난이도가 달랐다.

## 아직 풀리지 않은 문제

톤이 비슷해진다. 세 편을 연달아 쓰다 보니 문장 구조가 점점 닮았다. 같은 명령어 파일, 같은 금기어 목록, 같은 구조 지침을 쓰니까 당연한 결과다. 세 번째 글쯤 되면 읽으면서 "이 패턴 어디서 봤는데"가 느껴졌다.

지금 생각하는 해결 방향은 두 가지다.

첫째, 명령어 파일에 "이번 글의 문체 방향"을 인자로 받도록 바꾼다. `/blog-auto-write "주제" --tone=건조한-기록체` 식으로. 톤마다 예시 문장을 두세 개 붙여두면 편차가 생길 것 같다. 둘째, 최근 3편의 문장 구조를 읽어서 반복 패턴을 감지하고, 그 패턴을 피하도록 지시하는 검토 단계를 추가한다. 아직 구현 안 했고, 다음 주 과제다.

자동화가 글쓰기 자체를 대체하지는 않는다. 첫 줄을 넘기는 진입 비용을 낮춰줄 뿐이다. 그것만으로도 충분히 값어치가 있었다.
